# Tier Applicability Matrix

This page lists every solution in the catalog and shows which configuration tiers it ships with, plus the tier its README and architecture treat as the primary fit.

The data on this page is sourced from `data/solution-catalog.json` (`tiers_supported`, `tier_recommended`, `tier_maturity`). It is regenerated by `scripts/build-docs.py` from the same canonical source as `solutions.json` and `solutions-graph.json`, so it stays in lockstep with the catalog and the per-solution `config/baseline.json`, `config/recommended.json`, and `config/regulated.json` files.

> **Documentation-first scope.** The matrix describes which tier configs each solution ships with and which one its documentation is primarily organized around. It does not assert that any tier is operating in a customer tenant. See [Disclaimer](../disclaimer.md) and [Documentation vs Runnable Assets Guide](../documentation-vs-runnable-assets-guide.md).

## Legend

- **B / R / Reg** — Baseline / Recommended / Regulated tier configs.
- **Bold** — the tier that the solution README and architecture treat as the primary fit. Defaults: regulated for P0 solutions, recommended for P1/P2 solutions, unless a README explicitly indicates otherwise.
- ~~Strikethrough~~ — tier is not supported by this solution (no shipped config).
- `*` suffix — tier is supported but its config is a placeholder scaffold only (no representative settings yet).

All twenty-three solutions currently ship active configs for all three tiers.

## How to use this matrix

- Pick the **recommended** tier as the starting point when first deploying a solution. It reflects the configuration the documentation walks through end to end.
- The **baseline** tier supports lighter scope, smaller scan windows, and shorter evidence retention. It helps meet entry-level governance commitments while a tenant matures.
- The **regulated** tier helps meet the stricter examination-readiness, retention, and immutability expectations described in the regulatory mappings under each solution's `## Regulatory Alignment` section.
- The recommended tier is a documentation default, not a directive. Operators should choose the tier that matches their regulatory posture and risk appetite.
