#### Finding F4 â€” `SecurityEvents.Read.All` points to the deprecated legacy alerts API

- **Severity:** Major
- **Location:** lines 14-15
- **Current text:** "`SecurityEvents.Read.All` for Graph-based operational and security event enrichment where available"
- **Issue:** `SecurityEvents.Read.All` is documented for the deprecated legacy `/security/alerts` API. The prerequisite does not say it is legacy-only, and current Microsoft Graph security enrichment should specify the actual API and permission, such as `SecurityAlert.Read.All` for `/security/alerts_v2`, `SecurityIncident.Read.All` for `/security/incidents`, or `ThreatHunting.Read.All` for beta advanced hunting with an explicit preview caveat.
- **Recommended correction:** Replace the broad `SecurityEvents.Read.All` bullet with API-specific current permissions and note any beta/preview dependency.
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/alert-list?view=graph-rest-1.0
- **Source last updated:** 2026-03-05
- **Quote from source:** "The legacy alerts API is deprecated and will be removed by April 2026."

#### Finding F5 â€” Graph service-health polling should require Graph authentication tooling, not `Microsoft.Graph.Reports`

- **Severity:** Major
- **Location:** lines 26-32
- **Current text:** "`Microsoft.Graph.Reports`; `ExchangeOnlineManagement`; If the target operating model uses certificate-based authentication or broader Graph automation, install the supporting Microsoft Graph authentication module..."
- **Issue:** The live target for service health is a Microsoft Graph service communications REST endpoint, not an Exchange Online or Microsoft Graph reports cmdlet. Treating the Graph authentication module as conditional can leave implementers without `Connect-MgGraph`/`Invoke-MgGraphRequest` guidance for live service-health polling.
- **Recommended correction:** Require `Microsoft.Graph.Authentication` (or the full `Microsoft.Graph` SDK) for service-health polling; list `ExchangeOnlineManagement` only if an explicit optional Exchange-specific check is documented.
- **Source URL:** https://learn.microsoft.com/en-us/powershell/module/microsoft.graph.authentication/?view=graph-powershell-1.0
- **Source last updated:** 2026-03-02
- **Quote from source:** "Invoke-MgGraphRequest issues REST API requests to the Graph API. It works for any Graph API"

### `solutions/13-dora-resilience-monitor/scripts/Monitor-Compliance.ps1`

#### Finding F7 â€” Service-health status classification does not handle Microsoft Graph `serviceHealthStatus` values

- **Severity:** Critical
- **Location:** lines 123-126, 207-248
- **Current text:** The script defaults sample issues to `Degraded`, skips only `Healthy`, matches incident statuses with `Degraded|Advisory|Interruption|Investigating`, and marks resolved incidents with `Resolved|Restored`.
- **Issue:** Microsoft Graph service communications returns `serviceHealthStatus` values such as `serviceOperational`, `serviceDegradation`, and `serviceInterruption`. A Graph-shaped sample payload with `serviceOperational` would not match `Healthy` and could be incorrectly classified as an incident.
- **Recommended correction:** Normalize Graph status values before classification. Treat `serviceOperational`, `serviceRestored`, `verifyingService`, and `falsePositive` as non-open/healthy as appropriate; treat `serviceDegradation`, `serviceInterruption`, `investigating`, `restoringService`, and `extendedRecovery` as active issue states.
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/resources/servicehealth?view=graph-rest-1.0
- **Source last updated:** 2025-12-03
- **Quote from source:** "possible values are: `serviceOperational`, `investigating`, `restoringService`, `verifyingService`, `serviceRestored`"
