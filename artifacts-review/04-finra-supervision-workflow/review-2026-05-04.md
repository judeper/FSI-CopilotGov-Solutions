#### Finding F1 — Communication Compliance licensing uses former SKU name

- **Severity:** Major
- **Location:** line 53
- **Current text:** "Microsoft 365 E5 Compliance for Microsoft Purview Communication Compliance signals."
- **Issue:** Microsoft Learn now describes the license as Microsoft Purview Suite, formerly Microsoft 365 E5 Compliance, and also documents eligible Office 365 E5 / E3 plus Advanced Compliance options.
- **Recommended correction:** "Microsoft Purview Suite (formerly Microsoft 365 E5 Compliance), Office 365 Enterprise E5, or Office 365 Enterprise E3 with the Advanced Compliance add-on for Microsoft Purview Communication Compliance coverage."
- **Source URL:** https://learn.microsoft.com/purview/communication-compliance-plan
- **Source last updated:** 2025-08-12
- **Quote from source:** "Microsoft Purview Suite (formerly known as Microsoft 365 E5 Compliance) license"

#### Finding F2 — Communication Compliance admin role name is not current

- **Severity:** Major
- **Location:** line 54
- **Current text:** "Power Platform Admin, Purview Compliance Admin, and Global Reader access for deployment validation."
- **Issue:** "Purview Compliance Admin" is not the role/role-group name used by Microsoft Learn for Communication Compliance. Practitioners should assign either Microsoft Entra Compliance Administrator, Microsoft Purview portal Compliance Administrator, or the Communication Compliance / Communication Compliance Admins role groups, depending on scope.
- **Recommended correction:** Replace "Purview Compliance Admin" with "Communication Compliance Admins role group or Microsoft Entra Compliance Administrator / Microsoft Purview portal Compliance Administrator role group, as approved for least privilege."
- **Source URL:** https://learn.microsoft.com/purview/communication-compliance-permissions
- **Source last updated:** 2025-08-14
- **Quote from source:** "_Communication Compliance Admins_ role group"

#### Finding F3 — Ingestion flow describes an unsupported scheduled Purview polling pattern

- **Severity:** Major
- **Location:** lines 78-80 and 100
- **Current text:** "Trigger: scheduled poll or event-driven connector action that reads flagged Copilot prompt and response items from Microsoft Purview Communication Compliance."
- **Issue:** Microsoft Learn documents Power Automate in Communication Compliance as alert/user automation launched from an alert, not as a scheduled/event connector that polls flagged policy-match items from Communication Compliance.
- **Recommended correction:** Document that ingestion requires a customer-validated supported export/integration pattern, such as Communication Compliance report exports, audit-log review, or a Power Automate flow launched from a Communication Compliance alert; avoid implying an out-of-box scheduled polling connector.
- **Source URL:** https://learn.microsoft.com/purview/communication-compliance-power-automate
- **Source last updated:** 2025-04-07
- **Quote from source:** "working directly in an alert"

#### Finding F11 — Graph API permissions list wrong audit-search permission name

- **Severity:** Major
- **Location:** lines 34-41
- **Current text:** "Typical permissions... `Policy.Read.All`, `AuditLog.Read.All`, `User.Read.All`."
- **Issue:** Microsoft Graph Audit Search API uses `AuditLogsQuery-*` permissions, not `AuditLog.Read.All`. `Policy.Read.All` is not documented as an audit-search permission for `/security/auditLog/queries`.
- **Recommended correction:** For Audit Search Graph API scenarios, use `AuditLogsQuery.Read.All` or the least-privileged service-specific permissions such as `AuditLogsQuery-Entra.Read.All`, plus separate Graph permissions only where needed for user/group lookup.
- **Source URL:** https://learn.microsoft.com/graph/api/security-auditcoreroot-list-auditlogqueries?view=graph-rest-1.0
- **Source last updated:** 2024-10-29
- **Quote from source:** "AuditLogsQuery.Read.All"

### `scripts/Export-Evidence.ps1`

#### Finding F16 — Dataverse Web API queries use table logical names instead of EntitySet names

- **Severity:** Critical
- **Location:** lines 279 and 356-358
- **Current text:** "`$uri = '{0}/api/data/v9.2/{1}?$select={2}'`" with table names `fsi_cg_fsw_queue`, `fsi_cg_fsw_log`, and `fsi_cg_fsw_config`.
- **Issue:** Dataverse Web API collection paths require the environment's EntitySet resource name, which is usually plural and can differ from the table logical name. Using logical names directly can cause live export requests to fail.
- **Recommended correction:** Resolve EntitySet names from the Dataverse service document or make them explicit configuration values (for example, `fsi_cg_fsw_queues` if that is the actual EntitySet name), then call `/api/data/v9.2/{entitySetName}`.
- **Source URL:** https://learn.microsoft.com/power-apps/developer/data-platform/webapi/query/overview
- **Source last updated:** 2026-01-09
- **Quote from source:** "These values are usually the plural name of the table, but they can be different."
