## Findings

### F1 — Graph beta Pages enumeration and Microsoft Loop API claims are unsupported

- **Severity:** Critical
- **Location:** lines 11, 37, 49, 69-77
- **Current:** "Microsoft Graph (Copilot Pages, OneNote, Loop) - live target"; "Microsoft Loop APIs can expose Pages..."; "Microsoft Graph beta endpoints for Copilot Pages enumeration"
- **Issue:** The architecture identifies non-documented integration surfaces. Current Microsoft Learn content describes Copilot Pages/Notebooks storage in SharePoint Embedded and supports Graph API export for third-party tools; it does not document a Microsoft Graph beta enumeration endpoint for Copilot Pages or a standalone Microsoft Loop API for this purpose.
- **Suggested change:** Rewrite the integration model around documented surfaces: SharePoint Embedded containers and container URLs, Microsoft Graph DriveItem/export capabilities where documented, Purview audit/retention, and Cloud Policy/SharePoint admin controls. Do not reference Graph beta Pages enumeration unless Microsoft publishes a specific endpoint.
- **Source:** https://learn.microsoft.com/en-us/microsoft-365/loop/cpcn-compliance-summary?view=o365-worldwide (last updated 2026-04-28)
- **Quote:** "Graph API export: Supported for third-party tools."

### F2 — Branch/fork/mutability lifecycle is not documented for Copilot Pages

- **Severity:** Major
- **Location:** lines 5, 39, 42, 49, 59-61, 87
- **Current:** "mutability and branching lineage"; "Page branching, fork, and mutability transitions"
- **Issue:** Microsoft Learn documents Copilot Pages/Notebooks audit logs and version history, but not branch, fork, merge, or mutability-state events as product lifecycle signals.
- **Suggested change:** Replace the Branching Event Recorder with a version-history/audit-log evidence component, or explicitly mark branch/fork/mutability as repository sample taxonomy not backed by Microsoft product events.
- **Source:** https://learn.microsoft.com/en-us/microsoft-365/loop/cpcn-compliance-summary?view=o365-worldwide (last updated 2026-04-28)
- **Quote:** "Version History: Export in Purview or via Graph API... 50 versions per file (not configurable)."

### F3 — OneNote retention resolver is modeled at the wrong granularity

- **Severity:** Major
- **Location:** lines 40, 51-53, 71-72, 81
- **Current:** "retention label assigned to each Notebook"; "OneNote Notebook retention-label assignment"
- **Issue:** Microsoft 365 retention for OneNote content applies to sections as files, with pages inheriting from their parent section. Modeling direct notebook-level retention labels or notebook policy inheritance is inaccurate.
- **Suggested change:** Model OneNote sections/folders and parent SharePoint/OneDrive retention coverage; keep notebook metadata only for grouping and reporting.
- **Source:** https://learn.microsoft.com/en-us/purview/retention-policies-sharepoint (last updated 2026-04-30)
- **Quote:** "Pages from each section are contained within the file and inherit the retention settings from their parent section."

### F4 — Copilot Pages retention labels need limited-application caveats

- **Severity:** Major
- **Location:** lines 39, 49, 81
- **Current:** "records the retention-label assignment... for each Page"; "Resolve retention labels and policies that apply to Copilot Pages"
- **Issue:** The architecture omits important Microsoft Learn caveats: retention labels for Copilot Pages have limited manual application; they cannot be viewed or applied directly in Copilot Pages, and record/regulatory record labels cannot be manually applied in Copilot Pages or Loop.
- **Suggested change:** Add a caveat that Copilot Pages label evidence should be resolved via OneDrive/SharePoint/Loop/Purview-supported paths and distinguish retention policies from limited retention-label support.
- **Source:** https://learn.microsoft.com/en-us/microsoft-365/loop/cpcn-compliance-summary?view=o365-worldwide (last updated 2026-04-28)
- **Quote:** "Retention labels cannot be viewed or applied directly from a Copilot Page."
