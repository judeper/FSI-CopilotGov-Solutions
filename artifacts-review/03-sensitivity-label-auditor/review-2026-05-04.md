### `solutions/03-sensitivity-label-auditor/README.md`

#### Finding F1 — `assignSensitivityLabel` needs protected API caveat

- **Severity:** Major
- **Location:** line 136
- **Current text:** "The `assignSensitivityLabel` API requires the correct Microsoft 365 and Purview licensing, plus approved permissions for bulk application scenarios."
- **Issue:** The Microsoft Graph API is a protected SharePoint/OneDrive API and requires validations beyond permission consent. The README also omits the least-privileged Graph permissions and cloud availability caveat.
- **Recommended correction:** State that `driveItem: assignSensitivityLabel` is a protected API for SharePoint/OneDrive files, requires protected API validation plus `Files.ReadWrite.All` or `Sites.ReadWrite.All`, and is only documented as available in the global service.
- **Source URL:** https://learn.microsoft.com/graph/api/driveitem-assignsensitivitylabel?view=graph-rest-1.0
- **Source last updated:** 2026-04-09
- **Quote from source:** "Protected APIs require you to have more validations, beyond permission and consent, before you can use them."

#### Finding F2 — Label group migration schedule is unsupported by current Learn docs

- **Severity:** Major
- **Location:** line 139
- **Current text:** "Forced auto-migration runs December 2025 through May 2026 and is irreversible."
- **Issue:** Current Microsoft Learn guidance describes automatic migration only for specific cases and says migration is gradually rolling out. It does not support the fixed December 2025-May 2026 forced-migration window stated here.
- **Recommended correction:** Replace the fixed schedule with: "Label groups are replacing parent labels. Migration is irreversible; automatic migration applies in documented cases, and other tenants should migrate when the Microsoft Purview portal banner is available."
- **Source URL:** https://learn.microsoft.com/purview/migrate-sensitivity-label-scheme
- **Source last updated:** 2026-04-17
- **Quote from source:** "Migration is gradually rolling out, so migrating your parent labels to label groups might not yet be available for your tenant."

#### Finding F3 — Sensitivity label definitions are beta-only in Microsoft Graph

- **Severity:** Major
- **Location:** lines 38-39
- **Current text:** "Microsoft Graph permissions for label and workload data collection are approved."
- **Issue:** The solution elsewhere pairs label definition collection with Graph `v1.0`, but the `security/informationProtection/sensitivityLabels` list API is documented under Microsoft Graph beta. The README should disclose the beta dependency for label definition enumeration.
- **Recommended correction:** Add a prerequisite note that organization sensitivity label enumeration through `/security/informationProtection/sensitivityLabels` currently uses Microsoft Graph beta and should not be described as a production `v1.0` dependency.
- **Source URL:** https://learn.microsoft.com/graph/api/security-informationprotection-list-sensitivitylabels?view=graph-rest-beta
- **Source last updated:** 2024-04-04
- **Quote from source:** "APIs under the `/beta` version in Microsoft Graph are subject to change."

### `solutions/03-sensitivity-label-auditor/config/default-config.json`

#### Finding F1 — `graphApiVersion` incorrectly implies all label APIs are v1.0

- **Severity:** Critical
- **Location:** line 97
- **Current text:** `"graphApiVersion": "v1.0"`
- **Issue:** The solution's monitoring script uses `/security/informationProtection/sensitivityLabels` for sensitivity label definitions, but Microsoft Learn documents that API under Microsoft Graph beta. A single `v1.0` version setting can lead implementers to call a non-GA endpoint as if it were v1.0.
- **Recommended correction:** Split Graph version configuration by API surface, for example `graphApiVersion: "v1.0"` for drive item actions and `sensitivityLabelDefinitionsApiVersion: "beta"` with an explicit beta caveat.
- **Source URL:** https://learn.microsoft.com/graph/api/security-informationprotection-list-sensitivitylabels?view=graph-rest-beta
- **Source last updated:** 2024-04-04
- **Quote from source:** "APIs under the `/beta` version in Microsoft Graph are subject to change."

### `solutions/03-sensitivity-label-auditor/docs/architecture.md`

#### Finding F1 — SharePoint/OneDrive label data is not a `contentLabel` driveItem field

- **Severity:** Critical
- **Location:** lines 37-40 and 76-78
- **Current text:** "Graph drive items" with "contentLabel data" and "SharePoint and OneDrive drive items plus `contentLabel` metadata"
- **Issue:** Microsoft Graph documents sensitivity label extraction for SharePoint/OneDrive files through `driveItem: extractSensitivityLabels` and `sensitivityLabelAssignment`, not a `contentLabel` driveItem field. Implementing the architecture as written would point builders at the wrong Graph shape.
- **Recommended correction:** Replace `contentLabel` drive item language with `driveItem: extractSensitivityLabels` / `sensitivityLabelAssignment` for supported SharePoint and OneDrive file types, and cite the limitations for encrypted/unsupported files.
- **Source URL:** https://learn.microsoft.com/graph/api/driveitem-extractsensitivitylabels?view=graph-rest-1.0
- **Source last updated:** 2024-04-04
- **Quote from source:** "Extract one or more sensitivity labels assigned to a drive item and update the metadata of a drive item."

#### Finding F2 — Exchange Graph messages do not expose first-class `contentLabel` data

- **Severity:** Major
- **Location:** lines 38-40 and 76-79
- **Current text:** "Exchange messages | Graph mail items | contentLabel data" and "Exchange messages and associated label metadata"
- **Issue:** Microsoft Graph `message` resources don't document a first-class `contentLabel` or sensitivity-label assignment property. The architecture should not imply Exchange label coverage can be collected directly from `/users/{user-id}/messages` as `contentLabel` data.
- **Recommended correction:** Clarify the intended Exchange collection source, such as Purview audit/activity data, documented extended properties, or a tenant-approved Exchange/Purview export pattern, and state limitations explicitly.
- **Source URL:** https://learn.microsoft.com/graph/api/resources/message?view=graph-rest-1.0
- **Source last updated:** 2024-08-23
- **Quote from source:** "internetMessageHeaders ... can also contain custom message headers that hold app data for the message."

### `solutions/03-sensitivity-label-auditor/docs/prerequisites.md`

#### Finding F1 — Licensing names don't match current sensitivity-labeling service description

- **Severity:** Major
- **Location:** lines 5-6
- **Current text:** "Microsoft Purview Information Protection P2" and "Microsoft Purview Suite (formerly Microsoft 365 E5 Compliance)"
- **Issue:** Current Microsoft Learn licensing language for sensitivity labeling uses Microsoft 365 E5 and Microsoft 365 Information Protection and Governance / Microsoft Purview Suite combinations. `Microsoft Purview Information Protection P2` is not the documented SKU name.
- **Recommended correction:** Reword licensing prerequisites to reference Microsoft 365 E5/A5/G5, Microsoft 365 Information Protection and Governance, Microsoft Purview Suite, or AIP Plan 2 where applicable, and avoid presenting `Microsoft Purview Information Protection P2` as a SKU.
- **Source URL:** https://learn.microsoft.com/office365/servicedescriptions/microsoft-365-service-descriptions/microsoft-365-tenantlevel-services-licensing-guidance/microsoft-purview-service-description
- **Source last updated:** 2026-04-16
- **Quote from source:** "Sensitivity labeling, including automatic or policy‑based labeling, requires a Microsoft 365 E5 license or Microsoft 365 Information Protection and Governance (IPG)."

#### Finding F2 — Required roles omit Purview Information Protection role groups

- **Severity:** Major
- **Location:** lines 10-18
- **Current text:** "Compliance Administrator", "Security Administrator", "Global Administrator"
- **Issue:** Microsoft Learn lists Information Protection role groups and Sensitivity Label Administrator/Reader options for sensitivity labels. The current list is overbroad and omits least-privilege Purview roles; Global Administrator is not the recommended operating role.
- **Recommended correction:** Add Information Protection, Information Protection Admins, Information Protection Analysts/Investigators/Readers, Sensitivity Label Administrator, and Sensitivity Label Reader as appropriate; reserve Global Administrator for break-glass role assignment.
- **Source URL:** https://learn.microsoft.com/purview/get-started-with-sensitivity-labels
- **Source last updated:** 2026-04-16
- **Quote from source:** "For a read-only role, use Sensitivity Label Reader."

#### Finding F3 — Application permission for label enumeration is missing `.All`

- **Severity:** Major
- **Location:** lines 20-29
- **Current text:** `InformationProtectionPolicy.Read`, `Sites.Read.All`, `Files.Read.All`, `Mail.Read`
- **Issue:** For application access to organization sensitivity labels, Microsoft Graph documents `InformationProtectionPolicy.Read.All`, not `InformationProtectionPolicy.Read`. The current prerequisite can cause app-only label enumeration to fail.
- **Recommended correction:** Distinguish delegated and application permissions: delegated `InformationProtectionPolicy.Read`; application `InformationProtectionPolicy.Read.All`; drive label extraction least-privileged `Files.Read.All`; assign-label scenarios `Files.ReadWrite.All` or `Sites.ReadWrite.All`.
- **Source URL:** https://learn.microsoft.com/graph/api/security-informationprotection-list-sensitivitylabels?view=graph-rest-beta
- **Source last updated:** 2024-04-04
- **Quote from source:** "Application InformationProtectionPolicy.Read.All"

#### Finding F2 — SharePoint/OneDrive collection should call `extractSensitivityLabels`

- **Severity:** Major
- **Location:** lines 341 and 353
- **Current text:** Source endpoints list `'/drives/{drive-id}/items/{item-id}'` and notes say coverage uses drive item metadata and sensitivity label references.
- **Issue:** The documented Graph API for extracting assigned sensitivity labels from files is the `extractSensitivityLabels` action, returning `extractSensitivityLabelsResult` / `sensitivityLabelAssignment`. The current source endpoint list omits the action and may mislead implementers.
- **Recommended correction:** Use `POST /drives/{drive-id}/items/{item-id}/extractSensitivityLabels` (and equivalent site/user path forms) for supported SharePoint/OneDrive files, with `Files.Read.All` for read-only extraction.
- **Source URL:** https://learn.microsoft.com/graph/api/driveitem-extractsensitivitylabels?view=graph-rest-1.0
- **Source last updated:** 2024-04-04
- **Quote from source:** "POST /drives/{drive-id}/items/{item-id}/extractSensitivityLabels"

- **Location:** line 365
