## Findings by file

### `solutions\20-generative-ai-model-governance-monitor\README.md`

#### Finding F1 — Model inventory scope omits Microsoft Foundry and partner model families

- **Severity:** Major
- **Location:** lines 9-27
- **Current text:** "The Generative AI Model Governance Monitor (GMG) provides a documentation-first model risk management (MRM) monitoring framework for Microsoft 365 Copilot" and documents a "Copilot model inventory pattern."
- **Issue:** For an in-tenant generative AI model governance monitor, the documented inventory is limited to Microsoft 365 Copilot and doesn't cover Microsoft Foundry/Azure OpenAI deployments or partner/community model providers such as Anthropic and xAI that Microsoft Learn documents as available through Foundry.
- **Recommended correction:** Broaden the monitored inventory to include Microsoft Foundry (formerly Azure AI Foundry), Azure OpenAI/Foundry model deployments, and partner/community providers, with provider, model family, deployment type, region/cloud, owner, lifecycle status, and attestation freshness fields.
- **Source URL:** https://learn.microsoft.com/azure/ai-foundry/what-is-azure-ai-foundry
- **Source last updated:** 2026-04-29
- **Quote from source:** "Foundry gives you access to over 1,900 models from Microsoft, OpenAI, Anthropic, Mistral, xAI, Meta, DeepSeek, Hugging Face, and more."

#### Finding F2 — Content safety controls are absent from monitored capabilities and roadmap

- **Severity:** Major
- **Location:** lines 11-27 and 97-104
- **Current text:** The monitored capabilities list model inventory, validation scope, ongoing monitoring, and third-party due diligence, but no Azure AI Content Safety, Prompt Shields, groundedness, protected-material, or guardrail evidence.
- **Issue:** Microsoft Learn documents Azure AI Content Safety as the Microsoft service for detecting harmful user-generated and AI-generated content. A generative AI model governance monitor should account for content safety/guardrail configuration and evidence, especially for Azure OpenAI and Foundry-hosted model deployments.
- **Recommended correction:** Add a content safety/guardrails monitoring capability and roadmap item covering Azure AI Content Safety, Prompt Shields, groundedness/protected-material checks where applicable, and evidence of policy configuration, review cadence, and exceptions.
- **Source URL:** https://learn.microsoft.com/azure/ai-services/content-safety/overview
- **Source last updated:** 2025-09-16
- **Quote from source:** "Azure AI Content Safety is an AI service that detects harmful user-generated and AI-generated content in applications and services."

### `solutions\20-generative-ai-model-governance-monitor\docs\architecture.md`

#### Finding F1 — Architecture data flow only inventories Copilot records

- **Severity:** Major
- **Location:** lines 41-43 and 48-50
- **Current text:** "Monitor-Compliance.ps1 builds a representative sample inventory snapshot for Microsoft 365 Copilot, Copilot Chat, and Copilot Agents."
- **Issue:** The architecture excludes Microsoft Foundry/Azure OpenAI model deployments and partner/community provider models, even though Microsoft Learn now positions Foundry as the unified platform for agents, models, tools, monitoring, evaluations, and governance.
- **Recommended correction:** Add architecture components and data-flow steps for Foundry projects/resources, Azure OpenAI/Foundry model deployments, partner/community model sources, content safety/guardrail posture, and attestation freshness collection.

- **Source last updated:** 2026-04-29
- **Quote from source:** "Microsoft Foundry unifies agents, models, and tools under a single management grouping with built-in enterprise-readiness capabilities including tracing, monitoring, evaluations."

#### Finding F2 — Copilot interaction telemetry is mislabeled as Microsoft Graph audit logs

- **Severity:** Major
- **Location:** lines 64-70
- **Current text:** "Microsoft Graph audit logs — Copilot interaction telemetry to support output sampling"
- **Issue:** Microsoft Learn documents Copilot prompts and responses as captured in the Microsoft Purview unified audit log. The phrase "Microsoft Graph audit logs" points implementers toward Entra directory/sign-in audit APIs rather than Purview Audit / unified audit log records for Copilot and AI interactions.
- **Recommended correction:** Replace this integration point with "Microsoft Purview Audit / unified audit log (optionally accessed through Audit Search Graph API) — Copilot and AI interaction audit events to support sampling and investigation."
- **Source URL:** https://learn.microsoft.com/purview/ai-m365-copilot
- **Source last updated:** 2026-05-01
- **Quote from source:** "Like other activities, prompts and responses are captured in the unified audit log."

#### Finding F3 — Purview prompt/response review wording conflates labels and classifiers

- **Severity:** Minor
- **Location:** lines 68-70
- **Current text:** "Microsoft Purview — sensitivity-labeled prompt and response review"
- **Issue:** Microsoft Learn distinguishes data classification for finding sensitive data in prompts/responses from sensitivity labels applied to source content, citations, and newly created content. The current wording implies prompts/responses themselves are sensitivity-labeled.
- **Recommended correction:** Reword to "Microsoft Purview — AI interaction auditing, data classification for prompts/responses, sensitivity-label context for source/cited content, eDiscovery, and DLP review."
- **Source URL:** https://learn.microsoft.com/purview/ai-m365-copilot
- **Source last updated:** 2026-05-01
- **Quote from source:** "Sensitive information types and trainable classifiers can be used to find sensitive data in user prompts and responses."

### `solutions\20-generative-ai-model-governance-monitor\docs\evidence-export.md`

#### Finding F1 — Evidence schema assumes Microsoft-only products and providers

- **Severity:** Major
- **Location:** lines 5-19 and 52-64
- **Current text:** `modelName — Microsoft product name`, `modelProvider — vendor (Microsoft)`, and third-party due diligence evidence collected from Microsoft.
- **Issue:** Foundry model inventory can include Azure OpenAI models and selected models from other providers. The evidence schema doesn't capture provider, deployment type, region/cloud, model lifecycle/preview status, model version, or attestation freshness for non-Microsoft model deployments.
- **Recommended correction:** Add fields for model source (`microsoft365copilot`, `foundry`, `azureopenai`, `partner`), provider, model family/name/version, deployment type, region/cloud, lifecycle/preview/deprecation status, responsible AI/transparency references, content safety profile, and attestation review dates.
- **Source URL:** https://learn.microsoft.com/azure/ai-foundry/model-inference/concepts/models
- **Source last updated:** 2026-04-17
- **Quote from source:** "Models sold directly by Azure include all Azure OpenAI models and specific, selected models from top providers."

#### Finding F2 — Evidence package lacks content safety and guardrail artifacts

- **Severity:** Major
- **Location:** lines 36-64 and 77-85
- **Current text:** Evidence artifacts cover inventory, validation summary, ongoing monitoring, and third-party due diligence, but no content safety, Prompt Shields, groundedness, or protected-material evidence.
- **Issue:** Azure AI Content Safety provides APIs specifically for AI safety and prompt protection. Without an evidence artifact for these controls, the package is incomplete for governance of deployed generative AI models beyond Microsoft 365 Copilot.
- **Recommended correction:** Add a `content-safety-and-guardrails` artifact with content safety resource status, Prompt Shields/groundedness/protected-material settings where applicable, filter thresholds, review cadence, exceptions, and last validation timestamp.
- **Source URL:** https://learn.microsoft.com/azure/ai-services/content-safety/overview
- **Source last updated:** 2025-09-16
- **Quote from source:** "Content Safety provides several APIs for different moderation needs."

### `solutions\20-generative-ai-model-governance-monitor\docs\prerequisites.md`

#### Finding F1 — Prerequisites omit Azure account/resources needed for Foundry and Content Safety scope

- **Severity:** Major
- **Location:** lines 3-13
- **Current text:** "Microsoft 365 E3 or E5 tenant with Copilot licensing for the population in scope" and PowerShell/Pester requirements.
- **Issue:** The prerequisites only cover Microsoft 365 Copilot. If the solution governs in-tenant generative AI models across Microsoft Foundry, Azure OpenAI, partner models, and Content Safety, operators also need Azure account/subscription prerequisites, Foundry access, and Content Safety resource prerequisites.
- **Recommended correction:** Add prerequisites for Azure account/subscription, Microsoft Foundry access, Azure OpenAI/Foundry model inventory source access, Azure AI Content Safety resource/region, and any least-privilege Azure RBAC roles required for read-only inventory evidence.
- **Source URL:** https://learn.microsoft.com/azure/ai-foundry/what-is-azure-ai-foundry
- **Source last updated:** 2026-04-29
- **Quote from source:** "You need an Azure account. Then sign in to Microsoft Foundry."

#### Finding F2 — Role guidance lists overbroad/inexact Entra Global Admin requirement

- **Severity:** Major
- **Location:** lines 22-27
- **Current text:** "Entra Global Admin — required only when future versions add tenant integration"
- **Issue:** Current Microsoft Entra role documentation includes AI-specific built-in roles for Microsoft 365 Copilot and AI-related enterprise services. Listing only "Entra Global Admin" is imprecise, over-privileged for many governance review tasks, and doesn't use the exact role name "Global Administrator" if that role is truly required.
- **Recommended correction:** Replace with least-privilege role guidance such as AI Reader for review, AI Administrator for Microsoft 365 Copilot/AI administration, workload-specific Purview/Audit roles for audit evidence, Azure RBAC reader roles for Foundry resources, and "Global Administrator" only for tenant-wide consent or configuration that explicitly requires it.
- **Source URL:** https://learn.microsoft.com/entra/identity/role-based-access-control/permissions-reference
- **Source last updated:** 2026-04-29
- **Quote from source:** "AI Administrator Manage all aspects of Microsoft 365 Copilot and AI-related enterprise services in Microsoft 365."
