#### Finding F2 — Named location values are display-name placeholders, not Graph location IDs

- **Severity:** Major
- **Location:** lines 25-36 and 51-53
- **Current text:** "`Copilot-Zone-2`" and "`Copilot-Zone-3`" are used as `namedLocations` values.
- **Issue:** Microsoft Graph Conditional Access `includeLocations` expects location IDs or the special values `All`/`AllTrusted`. Passing display-name-like strings into generated Graph request bodies will not identify tenant named locations.

- **Source last updated:** 2024-07-22
- **Quote from source:** "Location IDs in scope of policy unless explicitly excluded, `All`, or `AllTrusted`."

### `solutions/07-conditional-access-automation/scripts/Deploy-Solution.ps1`

#### Finding F1 — Generated Graph policy uses named-location labels where IDs are required

- **Severity:** Major
- **Location:** lines 208-236
- **Current text:** "`includeLocations = if ($RiskTierSettings.namedLocationRequired) { $namedLocations } else { @('All') }`"
- **Issue:** `$namedLocations` is populated from config values such as `Copilot-Zone-2`, `Copilot-Zone-3`, or `NamedLocationReviewRequired`. Microsoft Graph expects named-location IDs, `All`, or `AllTrusted` in `includeLocations`.
- **Recommended correction:** Resolve configured named-location display names to Graph namedLocation IDs before building request bodies, and prevent `NamedLocationReviewRequired` from being emitted into executable Graph commands.
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/resources/conditionalaccesslocations?view=graph-rest-1.0
- **Source last updated:** 2024-07-22
- **Quote from source:** "Location IDs in scope of policy unless explicitly excluded, `All`, or `AllTrusted`."

#### Finding F2 — `blockLegacyAuth` selects modern client apps instead of legacy clients

- **Severity:** Critical
- **Location:** lines 227-232
- **Current text:** "When `$Configuration.blockLegacyAuth` is true, `clientAppTypes` is set to `browser` and `mobileAppsAndDesktopClients`."
- **Issue:** Browser and mobile/desktop clients are modern client app types; legacy authentication clients are Exchange ActiveSync and other clients. The regulated template can claim to block legacy authentication without applying to the legacy client types.
- **Recommended correction:** Generate a separate block policy for legacy authentication using `clientAppTypes` values such as `exchangeActiveSync` and `other`, with an explicit `block` grant control, or otherwise align the condition with Microsoft Learn legacy-client guidance.
- **Source URL:** https://learn.microsoft.com/en-us/entra/identity/conditional-access/concept-conditional-access-conditions
- **Source last updated:** 2026-04-01
- **Quote from source:** "Legacy authentication clients ... Exchange ActiveSync clients ... Other clients"

#### Finding F3 — Positive compliant-device filters can exclude noncompliant or unregistered devices from the policy

- **Severity:** Critical
- **Location:** lines 240-251
- **Current text:** "`deviceFilter` uses `mode = 'include'` with `device.isCompliant -eq True` or `device.trustType -eq \"ServerAD\"`."
- **Issue:** A positive include filter applies only when the device exists and matches the attribute. Noncompliant or unregistered devices can fall outside the policy scope, undermining the compliant-device requirement.
- **Recommended correction:** Use the `compliantDevice` grant control without narrowing the policy to compliant devices, and use separate block policies or negative device filters if the intent is to block unknown/nonmatching device states.
- **Source URL:** https://learn.microsoft.com/en-us/entra/identity/conditional-access/concept-condition-filters-for-devices
- **Source last updated:** 2026-03-24
- **Quote from source:** "If you were to use a positive operator, the filter rule would only apply when a device exists"
