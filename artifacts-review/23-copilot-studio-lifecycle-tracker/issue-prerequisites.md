## Findings

### F1 — Power Platform PowerShell module conflicts with PowerShell 7.2 requirement

- **Severity:** Major
- **Location:** lines 25-28
- **Current:** "PowerShell 7.2 or later" plus `Microsoft.PowerApps.Administration.PowerShell` for live integration.
- **Issue:** Microsoft Learn states the Power Apps/Power Platform admin PowerShell modules require Windows PowerShell 5.x and are incompatible with PowerShell 6.0 and later.
- **Suggested change:** Distinguish current scaffold scripts (PowerShell 7.2+) from legacy Power Platform admin module usage (Windows PowerShell 5.1), or recommend REST API/SDK integration for PowerShell 7+.
- **Source:** https://learn.microsoft.com/en-us/power-platform/admin/powerapps-powershell (last updated 2026-02-25)
- **Quote:** "The modules described in this document use .NET Framework, which is incompatible with PowerShell 6.0 and later."

### F2 — Live Power Platform API prerequisites omit current delegated-permission/RBAC model

- **Severity:** Major
- **Location:** lines 9-16, 20-21
- **Current:** "Power Platform admin API access" and client secret/certificate authentication.
- **Issue:** Current Power Platform API authentication guidance requires delegated permissions for user-context applications, and service-principal identities should use Power Platform RBAC roles rather than application permissions.
- **Suggested change:** Specify required Power Platform API permission scopes and, for service principals, required RBAC role assignment and preview caveats where applicable.
- **Source:** https://learn.microsoft.com/en-us/power-platform/admin/programmability-authentication-v2 (last updated 2026-03-09)
- **Quote:** "Power Platform API uses delegated permissions only at this time."

### F3 — Environment-scoped role name is not current

- **Severity:** Major
- **Location:** line 39
- **Current:** "Power Platform Administrator or environment-scoped Service Admin"
- **Issue:** "environment-scoped Service Admin" is not the documented environment role name; Microsoft Learn documents Environment Admin for environment-level administration.
- **Suggested change:** Use "Power Platform Administrator for tenant-wide administration, or Environment Admin for scoped environment administration" and add Copilot Studio/Dataverse role details separately.
- **Source:** https://learn.microsoft.com/en-us/power-platform/admin/wp-environments (last updated 2025-10-02)
- **Quote:** "The Environment Admin role can perform all administrative actions on an environment"

### F4 — Purview audit role requirement uses an imprecise role name

- **Severity:** Major
- **Location:** line 40
- **Current:** "Purview Compliance Admin"
- **Issue:** For Microsoft Purview audit search/review, Microsoft Learn specifies Audit Logs or View-Only Audit Logs roles. "Purview Compliance Admin" is not the precise documented role name.
- **Suggested change:** Replace with "Audit Logs or View-Only Audit Logs roles (for audit search), typically via Audit Manager/Audit Reader or an appropriate Microsoft Purview role group such as Compliance Administrator, depending on duties."
- **Source:** https://learn.microsoft.com/en-us/purview/audit-search (last updated 2026-02-18)
- **Quote:** "You must be assigned the Audit Logs or View-Only Audit Logs roles in the Microsoft Purview portal"
