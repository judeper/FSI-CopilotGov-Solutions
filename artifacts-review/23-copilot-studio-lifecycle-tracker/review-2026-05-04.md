## Findings by file

### `solutions/23-copilot-studio-lifecycle-tracker/README.md`

#### Finding F1 — Current Agent 365 lifecycle control plane is missing

- **Severity:** Major
- **Location:** lines 7-29, 101-107
- **Current text:** "The Copilot Studio Agent Lifecycle Tracker (CSLT) provides a documentation-first governance pattern for Microsoft Copilot Studio agents..." and the roadmap lists Power Platform admin API, Sentinel, and Power Automate integrations.
- **Issue:** As of the review date, Microsoft Agent 365 is a generally available Microsoft lifecycle, registry, and governance control plane for agents. The README frames lifecycle governance only around Copilot Studio/Power Platform artifacts and does not mention Agent 365, which can mislead practitioners evaluating the current Microsoft-native lifecycle surface.
- **Recommended correction:** Add Microsoft Agent 365 as a current integration/context item, clarify that CSLT supplements Microsoft Agent 365 and Copilot Studio evidence workflows, and note that Agent 365 is the Microsoft control plane for centralized agent registry/lifecycle governance where licensed.
- **Source URL:** https://learn.microsoft.com/en-us/microsoft-agent-365/overview
- **Source last updated:** 2026-05-01
- **Quote from source:** "Through the Agent 365 registry in the Microsoft 365 admin center, Microsoft Entra, and Microsoft Purview, admins can intentionally manage the lifecycle"

### `solutions/23-copilot-studio-lifecycle-tracker/docs/architecture.md`

#### Finding F2 — Architecture over-centers Power Platform/Copilot Studio APIs for lifecycle management

- **Severity:** Major
- **Location:** lines 11, 48, 58, 78-86
- **Current text:** "Power Platform Admin / Copilot Studio Management API" and "The Power Platform admin and Copilot Studio management surfaces can expose agent inventory, publishing events, and version history through documented APIs..."
- **Issue:** Current Microsoft Learn positions Microsoft Agent 365 as the centralized agent registry and lifecycle-governance plane across the Microsoft 365 admin center, Microsoft Entra, and Microsoft Purview. The architecture should not imply that Power Platform/Copilot Studio APIs alone are the current lifecycle-management surface for inventory, publishing, and version-history governance.
- **Recommended correction:** Add Microsoft Agent 365 registry/management as a first-class integration point and qualify Power Platform API usage as environment/resource metadata collection unless specific supported endpoints are documented.
- **Source URL:** https://learn.microsoft.com/en-us/microsoft-agent-365/overview
- **Source last updated:** 2026-05-01
- **Quote from source:** "centralizing lifecycle management, access control, and compliance across the enterprise. Through the Agent 365 registry"

### `solutions/23-copilot-studio-lifecycle-tracker/docs/prerequisites.md`

#### Finding F3 — Power Platform PowerShell module conflicts with PowerShell 7.2 requirement

- **Severity:** Major
- **Location:** lines 25-28
- **Current text:** "PowerShell 7.2 or later" and "`Microsoft.PowerApps.Administration.PowerShell` (when live Power Platform admin integration is added)"
- **Issue:** Microsoft Learn states the Power Apps/Power Platform admin PowerShell modules require Windows PowerShell 5.x and are incompatible with PowerShell 6.0 and later. Listing PowerShell 7.2 alongside that module will cause confusion or failed live-integration setup.
- **Recommended correction:** Distinguish current scaffold scripts (PowerShell 7.2+) from legacy Power Platform admin module usage (Windows PowerShell 5.1), or recommend the Power Platform REST API/SDK for PowerShell 7+ live integration.
- **Source URL:** https://learn.microsoft.com/en-us/power-platform/admin/powerapps-powershell
- **Source last updated:** 2026-02-25
- **Quote from source:** "The modules described in this document use .NET Framework, which is incompatible with PowerShell 6.0 and later."

#### Finding F4 — Live Power Platform API prerequisites omit current delegated-permission/RBAC model

- **Severity:** Major
- **Location:** lines 9-16, 20-21
- **Current text:** "Power Platform admin API access for Copilot Studio agent inventory" and "Client secret or certificate for app authentication"
- **Issue:** The current Power Platform API authentication guidance requires delegated permissions for user-context applications, and service-principal identities should use Power Platform RBAC roles rather than application permissions. The prerequisites do not call out the required API permission/RBAC model.
- **Recommended correction:** Specify the required Power Platform API permission scopes and, for service principals, the needed RBAC role assignment (for example Power Platform reader/contributor at the appropriate scope) and preview caveats where applicable.
- **Source URL:** https://learn.microsoft.com/en-us/power-platform/admin/programmability-authentication-v2
- **Source last updated:** 2026-03-09
- **Quote from source:** "Power Platform API uses delegated permissions only at this time."

#### Finding F5 — Environment-scoped role name is not current

- **Severity:** Major
- **Location:** line 39
- **Current text:** "Power Platform Administrator or environment-scoped Service Admin for Copilot Studio inventory access"
- **Issue:** Microsoft Learn documents tenant-level Power Platform service admin roles and environment-level roles such as Environment Admin. "environment-scoped Service Admin" is not the documented role name and may confuse implementers.
- **Recommended correction:** Use "Power Platform Administrator for tenant-wide administration, or Environment Admin for scoped environment administration" and add any required Copilot Studio/Dataverse role details separately.
- **Source URL:** https://learn.microsoft.com/en-us/power-platform/admin/wp-environments
- **Source last updated:** 2025-10-02
- **Quote from source:** "The Environment Admin role can perform all administrative actions on an environment"

#### Finding F6 — Purview audit role requirement uses an imprecise role name

- **Severity:** Major
- **Location:** line 40
- **Current text:** "Purview Compliance Admin for evidence review and retention oversight (when audit enrichment is added)"
- **Issue:** For Microsoft Purview audit search/review, Microsoft Learn specifies Audit Logs or View-Only Audit Logs roles, and Purview role groups include Audit Manager/Audit Reader and Compliance Administrator. "Purview Compliance Admin" is not the precise documented role name.
- **Recommended correction:** Replace with "Audit Logs or View-Only Audit Logs roles (for audit search), typically via Audit Manager/Audit Reader or an appropriate Microsoft Purview role group such as Compliance Administrator, depending on duties."
- **Source URL:** https://learn.microsoft.com/en-us/purview/audit-search
- **Source last updated:** 2026-02-18
- **Quote from source:** "You must be assigned the Audit Logs or View-Only Audit Logs roles in the Microsoft Purview portal"

### `solutions/23-copilot-studio-lifecycle-tracker/docs/deployment-guide.md`

#### Finding F7 — Deployment guidance should distinguish scaffold PowerShell from live Power Platform module support

- **Severity:** Major
- **Location:** line 5
- **Current text:** "The deployment script assumes Power Platform admin access for the live integration target, an Entra ID application registration where authentication is later wired in, and PowerShell 7.2 or later."
- **Issue:** This conflates the scaffold script runtime with future live Power Platform admin module integration. Microsoft Learn states the Power Apps/Power Platform admin PowerShell modules use .NET Framework and are incompatible with PowerShell 6.0 and later.
- **Recommended correction:** Clarify that repository scripts run in PowerShell 7.2+, but any live integration using `Microsoft.PowerApps.Administration.PowerShell` must use Windows PowerShell 5.1 or use REST API/SDK alternatives compatible with PowerShell 7+.
- **Source URL:** https://learn.microsoft.com/en-us/power-platform/admin/powerapps-powershell
- **Source last updated:** 2026-02-25
- **Quote from source:** "PowerShell in this article requires Windows PowerShell version 5.x."
