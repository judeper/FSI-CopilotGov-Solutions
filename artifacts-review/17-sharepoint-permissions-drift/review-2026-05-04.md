### `solutions/17-sharepoint-permissions-drift/DELIVERY-CHECKLIST.md`

#### Finding F1 — Global Reader/Security Reader is not sufficient for complete permissions inventory

- **Severity:** Critical
- **Location:** line 26
- **Current text:** "Global Reader or Security Reader role is confirmed for read-only operations"
- **Issue:** The checklist presents Global Reader or Security Reader as sufficient for read-only permission drift operations. Microsoft Graph's permissions API warns that non-owner callers only receive sharing permissions that apply to that caller, so this role guidance can produce incomplete permission data unless paired with appropriate SharePoint access or Microsoft Graph application permissions.
- **Recommended correction:** "For read-only drift detection, confirm SharePoint Administrator/site owner access or admin-consented Microsoft Graph application permissions such as `Files.Read.All`/`Sites.Read.All`; do not rely on Global Reader or Security Reader alone."
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/driveitem-list-permissions?view=graph-rest-1.0&tabs=http
- **Source last updated:** 2017-09-10
- **Quote from source:** "For a non-owner caller, only the sharing permissions that apply to the caller are returned."

### `solutions/17-sharepoint-permissions-drift/docs/prerequisites.md`

#### Finding F1 — Global Reader/Security Reader is not sufficient for complete permissions inventory

- **Severity:** Critical
- **Location:** line 17
- **Current text:** "Global Reader or Security Reader | Sufficient for read-only drift detection (no reversion)"
- **Issue:** This overstates what directory read roles can do for SharePoint permission enumeration. Microsoft Graph returns limited sharing-permission data to non-owner callers; complete read-only drift detection requires appropriate SharePoint access and/or admin-consented application permissions.
- **Recommended correction:** Replace the row with guidance such as: "SharePoint Administrator/site owner access or Microsoft Graph application permissions (`Files.Read.All` least privileged for drive item permissions; `Sites.Read.All` where site-wide inventory requires it) are required for complete read-only permission inventory. Global Reader/Security Reader alone is not sufficient."
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/driveitem-list-permissions?view=graph-rest-1.0&tabs=http
- **Source last updated:** 2017-09-10
- **Quote from source:** "For a non-owner caller, only the sharing permissions that apply to the caller are returned."

#### Finding F2 — Exchange Online Administrator is not required for Graph sendMail

- **Severity:** Major
- **Location:** line 18
- **Current text:** "Exchange Online Administrator | Required if using Graph API mail notifications"
- **Issue:** Microsoft Graph `sendMail` requires `Mail.Send` permissions for delegated or application use; the Learn permission table does not require Exchange Online Administrator. Keeping this role as required overstates administrator privilege needs.
- **Recommended correction:** Remove the Exchange Online Administrator prerequisite. Document `Mail.Send` admin consent, a licensed sender mailbox, and any application access policy or mailbox scoping requirements used by the institution.
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/user-sendmail?view=graph-rest-1.0&tabs=http
- **Source last updated:** 2024-06-21
- **Quote from source:** "Application Mail.Send Not available."

#### Finding F3 — `User.Read.All` does not cover group identity resolution

- **Severity:** Major
- **Location:** line 48
- **Current text:** "`User.Read.All` | Application | Resolve user and group identities"
- **Issue:** `User.Read.All` can support user lookups, but group enumeration/resolution requires a group or directory permission. The current permission table omits the Graph permissions needed for group identities.
- **Recommended correction:** Change the row to separate user and group requirements, for example: "`User.Read.All` for user resolution; `GroupMember.Read.All`/`Group.Read.All` or `Directory.Read.All` for group resolution, depending on the queries used."
- **Source URL:** https://learn.microsoft.com/en-us/graph/api/group-list?view=graph-rest-1.0&tabs=http
- **Source last updated:** 2024-10-15
- **Quote from source:** "Application GroupMember.Read.All Directory.Read.All, Directory.ReadWrite.All, Group.Read.All, Group.ReadWrite.All"

#### Finding F4 — Azure AD branding is stale

- **Severity:** Major
- **Location:** lines 41, 64
- **Current text:** "register an Azure AD application"; "Azure AD authentication"
- **Issue:** Microsoft renamed Azure Active Directory (Azure AD) to Microsoft Entra ID. The authentication endpoint can remain `login.microsoftonline.com`, but the product name should be current.
- **Recommended correction:** Use "Microsoft Entra ID application registration" and "Microsoft Entra ID authentication" while leaving `login.microsoftonline.com` unchanged.
- **Source URL:** https://learn.microsoft.com/en-us/entra/fundamentals/new-name
- **Source last updated:** 2025-05-07
- **Quote from source:** "The names Azure Active Directory, Azure AD, and AAD are replaced with Microsoft Entra ID."

#### Finding F5 — Azure Automation PowerShell runtime guidance is outdated

- **Severity:** Major
- **Location:** line 37
- **Current text:** "Azure Automation environments are limited to PnP.PowerShell 2.12.0 (PowerShell 7.2 only)."
- **Issue:** Current Microsoft Learn guidance says Azure Automation supports PowerShell 7.4 and 5.1, and recommends PowerShell 7.4 because PowerShell 7.2 is no longer supported by the parent product. The current sentence can incorrectly steer implementers away from supported 7.4 runtime environments.
- **Recommended correction:** Update the note to say Azure Automation supports PowerShell 7.4 and 5.1, with region/cloud caveats, and validate PnP.PowerShell module compatibility in the selected runtime.
- **Source URL:** https://learn.microsoft.com/en-us/azure/automation/automation-runbook-types
- **Source last updated:** 2025-07-15
- **Quote from source:** "The currently supported versions are PowerShell 7.4 and PowerShell 5.1."

#### Finding F1 — Sample evidence uses non-canonical SharePoint permission level name

- **Severity:** Minor
- **Location:** lines 154, 173
- **Current text:** "permissionLevel = 'FullControl'"
- **Issue:** Microsoft Learn documents the default SharePoint permission level as "Full Control" with a space. Sample evidence that uses `FullControl` can diverge from real SharePoint display values and confuse downstream review.
- **Recommended correction:** Use `Full Control` in sample evidence, or normalize permission levels with a clear mapping before export.
- **Source URL:** https://learn.microsoft.com/en-us/sharepoint/understanding-permission-levels
- **Source last updated:** 2025-04-30
- **Quote from source:** "Full Control Contains all available SharePoint permissions."
